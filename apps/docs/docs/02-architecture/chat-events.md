# Чат-события и персистентные эффекты

Этот документ фиксирует схему для долгоживущих эффектов (TTL) и доставки чат‑ивентов. Это план vNext и не ломает текущий MVP.

## Зачем это нужно

- Эффекты должны переживать перезагрузку клиента и краткий даунтайм.
- UI обязан показывать оставшееся время и статус.
- Чат‑ивенты должны доставляться надёжно и идемпотентно.

## Хранилище эффектов (Redis TTL)

Источник истины — Redis ключи с TTL. База фиксирует лишь историю и аналитические события, а live‑состояние держится в Redis.

Рекомендуемый шаблон ключа:

`effects:{room_id}:{target_user_id}:{effect_type}:{effect_id}`

Минимальное содержимое значения (JSON):

- `effect_id`
- `effect_type`
- `source_user_id`
- `target_user_id`
- `room_id`
- `created_at`
- `expires_at`
- `payload`

TTL выставляется по `expires_at` и гарантирует автоматическую очистку.

## Очередь чат‑ивентов

События создаются на сервере, кладутся в очередь и доставляются подписчикам комнаты.

Рекомендуемый контракт очереди (схема события):

- `event_id`
- `event_type` (например: `effect_applied`, `effect_expired`, `system_notice`)
- `room_id`
- `source_user_id`
- `target_user_id`
- `created_at`
- `payload`

Поддерживаемая семантика:

- Идемпотентность по `event_id`
- Повторы допустимы, UI должен игнорировать дубли
- Очередь хранит события минимум 24 часа

## Поток обработки

1. API проверяет право и валидирует контр‑связь.
2. API пишет эффект в Redis с TTL.
3. API отправляет `effect_applied` в очередь.
4. Воркеры публикуют событие в реальном времени (WebSocket/SSE).
5. По истечении TTL публикуется `effect_expired`.

## Поведение UI

- Эффекты читаются из live‑состояния и кэшируются локально.
- При получении события UI обновляет таймеры без перезагрузки комнаты.
- При конфликте локального и серверного времени приоритет — сервер.

## Безопасность

- Авторизация обязательна для каждого эффекта.
- Нельзя применять эффект к себе, если тип не допускает `self`.
- Любое применение логируется в базу как аудит‑событие.
